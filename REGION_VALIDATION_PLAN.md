# Region-Validation Plan for the X-Wines Import

## What this is, and why it belongs in this initiative

This plan replaces the two failed region-validation attempts (`region_authority` string-matching, and the never-tried `GEOGRAPHY` tree) with a single combined-signal design for the one remaining blocker on X-Wines import: `RegionName` on the 100,645 real rows going into `wine_knowledge`. It belongs here because import is explicitly blocked on this — nothing else in Phase 2 can go live until this is real, not because it's a nice-to-have safeguard.

## Objective — stated honestly, not overclaimed

This plan does not produce a global, ground-truth-verified `RegionName` for all 100,645 rows. No data source available covers 62 countries. What it does produce, for every row: a validation status (`CONFIRMED`, `CORRECTED`, `FLAGGED_CONFLICT`, or `UNVALIDATED`) with a recorded reason and source; real corrections only where an independent anchor justifies them, never a guess or raw majority vote; and an honest `UNVALIDATED` flag everywhere no reference data exists.

## New findings verified directly (not just the prior briefing)

- The `GEOGRAPHY` tree would ALSO have missed Albinea Canali: no Italian-language alias for "Piemonte", and "emilia-romagna" (the correct answer) isn't even a `GEOGRAPHY` key. Wiring it in naively wouldn't have worked either.
- Naive same-producer majority vote is actively dangerous: Albinea Canali's real 6 rows are 5 "Piemonte" vs. 1 correct — majority vote would flip the one right row to match the five wrong ones.
- Most within-producer disagreement in this dataset (checked 15 real cases) is granularity, not error — e.g. "Pessac-Léognan" vs. "Bordeaux" for the same château. A blanket "flag all disagreement" rule would flood review with false positives.
- A real, already-built, currently-unused path actually resolves the flagship case: `deriveItalyRegionFromMunicipalities()` fed from eAmbrosia + the real ISTAT municipality join. Computed real similarity between X-Wines' spelling and eAmbrosia's real PDO name: 0.839 — clears the existing 0.8 threshold.
- Real coverage, computed directly from the CSV: `GEOGRAPHY`'s 8 countries cover 77.7% of rows; eAmbrosia/municipality countries cover 64.4%; the union covers 86.9%. ~13% of rows have no reference data at all, in any source.
- Real producer-conflict counts: 30,862 (winery, country) groups; 6,862 disagree internally; only 984 have a clean majority — a narrow signal, not a broad one.

## Architecture: four signals, combined, never a silent guess

- **Signal A** — does this region exist at all for this country (`GEOGRAPHY`, `region_authority`, or eAmbrosia/municipality join)?
- **Signal B** — hierarchy check: is a same-producer disagreement a parent/child pair (compatible, leave alone) or two unrelated places (incompatible, real conflict)?
- **Signal C** — producer-group consistency, used only to detect a genuine conflict exists, never to pick a winner by vote.
- **Signal D** — independent anchor: a specific derived parent region tied to one row's exact text (eAmbrosia/municipality strongest, `region_authority` weaker).

Decision per producer group: no conflict → confirmed/unvalidated as found; conflict with an anchor → anchor confirmed, disagreeing rows corrected to it; conflict with no anchor → flagged for human review, untouched; no reference data for that country → unvalidated.

## Schema change needed first

Add to `wine_knowledge`: `regionValidationStatus` (enum), `regionOriginalValue` (audit trail, never overwritten), `regionValidationSource`, `regionFlagReason`. Same "keep the original, never silently overwrite" pattern already used for `sourceTrust`.

That cross-row conflict detection is Signal B/C, Step 4, not this step. `tsc --noEmit` clean. One-off verification script deleted after use, per its own header comment.
4. **DONE 2026-07-21.** Built `checkProducerGroupConsistency(countryCode, regionNameCounts)` in `lib/wines/region-hierarchy-checker.ts`. Design changed twice from the original Signal B framing after real testing disproved the first two versions, not assumed correct up front:
   - Real counter-example (Ken Wright Cellars: Willamette Valley Pinot Noir + Celilo Vineyard Chardonnay from White Salmon, Washington) proved hierarchy-relatedness can't gate flagging -- unrelated values can be completely legitimate. So the gate is an independent anchor (the eAmbrosia bridge), not geographic relatedness.
   - Running the first anchor-gated version against the real regression set surfaced two more real false positives: Fayolle Fils & Fille ("Crozes-Hermitage" wrongly corrected away from "Northern Rhône", an informal term with no eAmbrosia entry) and Terra d'Alter ("Alentejo" wrongly corrected to "Alentejo Central" relative to "Alentejano", which itself only scores 0.800 against eAmbrosia -- not confident). Fix: the majority value must **also** confidently self-anchor (>=0.85) before any minority value can be corrected or flagged against it -- an unanchored or unconfident majority means there's no trustworthy ground truth to judge anything against, so the whole group is left alone (`MAJORITY_UNANCHORED`).
   - A "clear majority" (one value with a strict majority, >50%, of the group's rows) is required before attempting anything at all; groups without one are presumed genuine multi-region producers (confirmed real cases: Quinta da Barreira/Douro+Lisboa, Lionel Osmin & Cie's many Southwest France appellations) and left alone (`NO_CLEAR_MAJORITY`) regardless of anchor results.
   - Threshold bands calibrated against real data, not picked arbitrarily: ran the eAmbrosia bridge against all 1,483 distinct (country, RegionName) values in eAmbrosia-covered countries and hand-inspected the borderline range. Below 0.75: reliably unrelated names. 0.75-0.84: a real mix of wanted matches (Albinea Canali, 0.839) and genuine different-place near-misses (e.g. "Tarantino"/"Trentino" 0.778) -- flagged for human review, not auto-corrected. >=0.85: spot-checked matches all clean. `EAMBROSIA_MATCH_FLOOR = 0.75`, `EAMBROSIA_AUTOCORRECT_FLOOR = 0.85`.
   - Two real bugs fixed in the underlying eAmbrosia bridge itself while calibrating: (1) eAmbrosia's PDOnam field bundles alternate spellings with "/" for 122 of 1,177 PDOs (e.g. "Erbaluce di Caluso / Caluso") -- wasn't being split for matching the way Municip_nam already was, now fixed. (2) Sub-zone/quality qualifiers ("Villages", "Grand Cru", "Classico", "Riserva", "Superiore") were dragging down otherwise-correct matches (e.g. "Amarone della Valpolicella Classico" only scored 0.743 against "Amarone della Valpolicella") -- now stripped symmetrically from both sides before scoring.
   - **Verified against the full locked 21-case regression set, end-to-end through the real code, not by hand:** 0 unexpected results. Albinea Canali correctly `FLAGGED_CONFLICT` (0.839, below the auto-correct floor -- a deliberate change from the original plan's expectation of an auto-`CORRECTED` outcome, made because the real data showed 0.75-0.89 is too noisy a band to silently auto-correct on). All 20 "leave untouched" cases correctly stay untouched, including the two that exposed the majority-anchoring bug. `tsc --noEmit` clean.
5. **DONE 2026-07-21.** Built `validateRegionBaseline()` in `lib/wines/region-validate-import.ts` and wired it, plus `checkProducerGroupConsistency()`, into `scripts/import-xwines.ts` as a real two-pass pipeline (Pass 1: per-row baseline; Pass 2: producer-group consistency, escalating Pass 1's CONFIRMED/UNVALIDATED to CORRECTED/FLAGGED_CONFLICT only where the group check catches something Pass 1 alone couldn't). A real design bug was found and fixed during this step, not before it shipped: the first version of `validateRegionBaseline` compared eAmbrosia's *derived broader region* (e.g. "Bourgogne-Franche-Comté", the admin area a matched PDO's municipalities sit in) directly against the source `RegionName` text (typically a specific appellation, e.g. "Chablis") and treated any non-agreement as a correction. Run against the real, full 64,863 eAmbrosia-covered rows, that produced 37,663 false "corrections" and 834 false "flags" -- caught only because this step included a full-catalog run, not just the earlier 12-row spot check. Root cause: Signal A is defined in this plan's own architecture section as existence-only ("does this region exist at all for this country"), never a per-row correction -- correction is exclusively a producer-group-conflict decision (Signal B/C). Fixed by making `validateRegionBaseline` existence-only (CONFIRMED if recognized by eAmbrosia or region_authority, UNVALIDATED if not); all CORRECTED/FLAGGED_CONFLICT status now comes only from `checkProducerGroupConsistency`, per the plan's own decision rule in the Architecture section above. Also added memoization to `deriveRegionFromEambrosia` (keyed on `countryCode::regionNameFromSource`) after timing the real run: a handful of large French négociants have 30-63 distinct RegionName values each, and the uncached version took ~80s across the full catalog's 6,862 disagreeing producer groups; memoized, the same real run takes ~11s.

This gap is now closed -- the numbers above are real, not an estimate.
   One nuance worth flagging plainly, seen directly in Amanda's real output: a handful of the 984 "corrected" rows show the same text before and after (e.g. "Douro" -> "Douro"). Not a bug -- by design, `checkProducerGroupConsistency` never rewrites a producer's *majority* claim (that's the whole point of the Ken Wright/Albinea Canali fix), so when a *minority* row's own stated value already matches what it independently anchors to, the correction is really "this row's claim is verified correct against a majority that may itself be wrong" -- the status changes, the text doesn't, because there was nothing to fix in that specific row's own text. The real, practical implication: a producer's *majority* rows being wrong (the original Albinea Canali problem) is never silently auto-fixed by this design -- only ever flagged, and only when the minority anchor lands in the uncertain 0.75-0.84 band. A majority that's wrong with no minority disagreement at all, or with a minority that itself lands below 0.75 (no anchor), stays invisible to this whole check, exactly as the "Residual gaps" section below already says.
